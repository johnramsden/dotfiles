Save a document to the Obsidian vault. Use when the user asks to write, add, or save an exploration, explanation, deep-dive, planning, or review document about: $ARGUMENTS

Follow these steps exactly:

1. Determine the repository name:
   - Run `git remote get-url origin 2>/dev/null | sed 's|.*/||; s|\.git$||'` to get it from git.
   - If that is empty, fall back to the basename of the current working directory.
   - If $ARGUMENTS contains an explicit repo name (e.g. "for ceph-csi"), use that instead.

2. Get today's date: run `date +%Y-%m-%d`.

3. Determine the document type from $ARGUMENTS or context, and map it to a subdirectory:
   - Exploration, explanation, deep-dive → `explanation/`
   - Planning, design, proposal → `planning/`
   - Review, audit, analysis → `reviews/`

4. Derive a short, descriptive kebab-case slug from the topic in $ARGUMENTS (e.g. "rados-object-lifecycle").

5. Construct the output path:
   ~/Canonical/Ceph/Docs/Ceph/repositories/<repo>/<subdir>/<YYYY-MM-DD>-<slug>.md

6. Create the directory if it does not exist, then write the document there.

The document must:
- Begin with a top-level `# Title` heading that matches the topic.
- Include a short introductory paragraph summarising the scope.
- Be organized into logical sections with `##` headings.
- Favour concrete detail, diagrams (as ASCII or Mermaid), and examples over vague prose.
- In Mermaid graph node labels, use `<br>` for line breaks — never `\n` (Obsidian does not render `\n` inside node label strings).
- In Mermaid sequence diagram message labels and notes, use `<br/>` for line breaks — never `\n` or shell-style backslash continuation.
- In Mermaid, never put `(`, `)`, `{`, or `}` inside node labels, edge labels, participant aliases, or sequence message/note text — Mermaid treats them as shape/syntax delimiters and they break rendering in Obsidian even inside quoted strings. Rephrase using plain words, commas, or em dashes (e.g. `config 0,1,2` not `config {0,1,2}`; `leader 10.0.0.1` not `leader (10.0.0.1)`).
- Use Markdown format.
- Not include a "Generated by Claude" footer or similar.

- In Mermaid, do not use `;` inside sequence-diagram message or note text — Mermaid treats `;` as a statement separator, so the message ends early and the rest becomes an invalid statement. Use a comma, period, or `<br/>` instead.

After writing the document, you MUST truly validate every Mermaid block by rendering it with the real Mermaid parser — visual inspection and grepping are not sufficient (e.g. a stray `;` passes a grep but fails the parser). Run:

```
~/.claude/scripts/validate-mermaid.sh <path-to-the-document>
```

- This script extracts each ```mermaid block and renders it with `mmdc` (`@mermaid-js/mermaid-cli`), printing the exact parse error and line for any block that fails. It exits non-zero if any block is invalid.
- If `mmdc` is not installed, install it with `sudo snap install mermaid-cli` (provides `/snap/bin/mmdc`) or `npm install -g @mermaid-js/mermaid-cli`, then re-run.
- Gotcha: the snap build of `mmdc` is confined — it cannot read `/tmp` or hidden (dotfile) paths, only non-hidden paths under `$HOME`. The script already stages its work in a non-hidden `$HOME` dir to satisfy this.
- If any block fails, fix it and re-run the script until it reports PASS for every block (exit 0) before reporting the task done.
